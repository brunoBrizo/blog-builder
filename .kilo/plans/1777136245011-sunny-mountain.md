## Detailed Findings

### 1. Architecture & Feature Coverage

- **Topic queue table** (`topic_queue`) correctly separates topic lifecycle from job lifecycle with `normalized_topic` unique index, score, status (`available`/`reserved`/`consumed`).
- **Scheduled trigger path** in controller: no `topicSeed` → `ensureMinDepth` → `reserveScheduledTopicAndCreateJob` under advisory lock (`pg_advisory_xact_lock`). Retry once if first reserve returns null. Correct.
- **Auto-publish**: `autoPublish` defaults `true` for scheduled, `false` for manual. Orchestrator `markSucceeded` checks `job.autoPublish && job.articleId` → publishes article, marks topic consumed, triggers ISR revalidation.
- **Retry flow**: `POST /internal/articles/generation/retry` lists `transient` failed jobs with `retryAfter <= now`, resets to `pending`, increments `retryAttempt`, re-emits Inngest event.
- **Failure classification**: `mapToNonRetriable` covers `KillSwitchError`, `BudgetExceededError`, `PerplexityValidationError`. Everything else (including Perplexity/network errors) is retriable by Inngest first, then classified `transient` in `onFailure` / `recordInngestExhaustedFailure`.
- **ISR revalidation**: Web route `POST /api/revalidate` validates `x-revalidate-secret`, calls `revalidatePath`/`revalidateTag`. API service `WebIsrRevalidationService` builds locale-prefixed blog paths and calls the web origin.

### 2. Critical Issues Found

#### Issue A: `WEB_PUBLIC_ORIGIN` required in production but not enforced in `env.schema.ts`

- The `env.schema.ts` superRefine block does validate `WEB_PUBLIC_ORIGIN` in production, but only via `!data.WEB_PUBLIC_ORIGIN || data.WEB_PUBLIC_ORIGIN.length === 0`.
- The schema default is `''` (empty string). The superRefine check correctly blocks empty strings in production. **OK**.
- However, `AppConfigService.webPublicOrigin` returns `''` when unset. `WebIsrRevalidationService.revalidateAfterPublish` throws in production if `origin.length === 0`. This is a runtime guard, but the env schema already prevents this at boot. **Minor duplication, not a bug.**

#### Issue B: Score stored as `numeric` in DB but typed as `string` in shared schema

- `libs/db/src/schema/topic-queue.ts`: `score: numeric(...)` with default `'0'`.
- `libs/shared-types/src/schemas/topic-queue.ts`: `score: z.string()`.
- **Risk**: Drizzle `numeric` returns `string` by default (correct for Postgres numeric). The shared schema matches this. **OK**.

#### Issue C: `revalidateAfterPublish` does not include category pages, homepage, or sitemap

- Feature spec requires revalidating: English article URL, translated URLs, homepage per locale, category page per locale, sitemap.
- Current implementation only adds `/${loc}/blog` and `/${loc}/blog/${slug}`.
- **Gap**: missing homepage (`/${loc}`), category pages (e.g., `/${loc}/blog` is list, not category — categories are separate), and sitemap (`/sitemap.xml`).
- **Severity**: Medium — acceptance criteria say "homepage per locale, category page per locale, and sitemap" should be revalidated.

#### Issue D: `topicQueue` enum not defined in Drizzle enum file

- `topicQueue.status` is `text('status')` with app-managed values (`available`, `reserved`, `consumed`).
- Feature spec does not strictly require a Postgres enum for this. Using text with app-level checks is acceptable and more flexible. **OK**.

#### Issue E: `onFailure` handler in `generate-article.function.ts` does not check `triggerKind`

- The `onFailure` handler calls `recordInngestExhaustedFailure` for all jobs. This is fine because manual jobs that exhaust retries should also be marked failed. However, `retryAfter` is set for all, meaning manual jobs could be picked up by the daily retry cron too.
- `recordInngestExhaustedFailure` sets `failureClass: 'transient'`. The retry endpoint does not filter by `triggerKind`. So a manually triggered job that fails after Inngest retries could be auto-retried by the cron retry job.
- **Severity**: Medium — manual jobs should probably not be auto-retried by the scheduler. The spec says "a separate pg_cron job once a day re-examines recent failed jobs and re-enqueues them". It does not explicitly exclude manual jobs, but the intent seems focused on scheduled pipeline failures.

#### Issue F: `reserveScheduledTopicAndCreateJob` does not dedupe against existing article slugs

- Feature spec: "Select the highest-scored topic whose normalized form does not match any articles.slug across locales, nor any recently failed topic."
- Current query: `where eq(topicQueue.status, 'available')` only. It does NOT check against `articles.slug` or `articleTranslations.slug`, nor against recently failed jobs.
- **Severity**: High — this is an explicit acceptance criterion: "No duplicate slug across locales and no duplicate topic across runs within 90 days."
- The `normalizedTopic` dedupe in `insertTopicQueueCandidates` uses `onConflictDoNothing`, but if an article was created manually (not via topic queue) with the same slug, the queue could still select a matching topic.

#### Issue G: `insertTopicQueueCandidates` score conversion

- `scoreVal = String(t.score ?? 0)` then inserted into `numeric`. If `t.score` is `undefined`, `0` is correct. OK.
- But the sort in `reserveScheduledTopicAndCreateJob` is `desc(topicQueue.score)`. Since `score` is numeric, `desc` works on the numeric type. OK.

#### Issue H: `PostInternalArticlesGenerateSchema` transforms empty string to undefined

- `z.preprocess((v) => (v === '' ? undefined : v), z.string().min(1).max(500).optional())`.
- This means `topicSeed: ''` becomes `undefined`, triggering scheduled mode. This is correct and safe. OK.

#### Issue I: `generate-article.function.ts` event typing

- `autoPublish?: boolean` is destructured from `event.data` but never passed into the pipeline/orchestrator explicitly. The orchestrator reads `job.autoPublish` from the DB after `markSucceeded`.
- This is actually the correct design: the controller writes `autoPublish` to the DB job row, and the pipeline executes independently. The orchestrator checks the DB flag at the end. OK.

#### Issue J: `topicQueueService.ensureMinDepth` does not check for duplicate slugs before inserting

- When refilling the queue, it blindly inserts research topics. The `onConflictDoNothing` on `normalizedTopic` handles duplicate topics within the queue, but not duplicate slugs against existing articles.
- Related to Issue F.

#### Issue K: Web build includes `/api/revalidate` but no method restrictions

- The route only handles `POST` in the handler function. Next.js App Router will return 405 for other methods automatically? Actually, Next.js route handlers don't auto-405 for undefined methods — they just return 404. This is acceptable. OK.

#### Issue L: Migration safety

- Migration uses `IF NOT EXISTS` / `DO` blocks for enum, columns, table, indexes, FKs, triggers. Safe to re-run. OK.
- `_journal.json` updated with new migration entry. OK.

### 3. Minor / Style Issues

- `article-generation.controller.ts` line 42: `const autoPublish = body.autoPublish ?? (isScheduled ? true : false);` — could be simplified to `body.autoPublish ?? isScheduled` but explicit is fine.
- `web-isr-revalidation.service.ts` line 46: `JSON.stringify({ paths: [...paths] })` — `paths` is already a Set spread into array, fine.
- `revalidate.ts` (web lib) now imports `revalidatePathAllowed` from shared-types and sends batch `{ paths: [path] }`. Consistent with new schema. OK.

### 4. Test Coverage

- `article-generation.pipeline.spec.ts` tests happy path, crash-resume, budget exhaustion, kill-switch. All pass.
- **Missing tests**: no tests for scheduled trigger path (no topic seed), auto-publish path, topic queue dedupe, retry endpoint, ISR revalidation route, or `reserveScheduledTopicAndCreateJob` concurrency.
- The in-memory repo implements all new methods but many are stubs (e.g., `countAvailableTopics` returns `0`, `insertTopicQueueCandidates` returns `0`). This means unit tests cannot meaningfully test topic queue refill.

## Recommendations (in priority order)

1. **Fix Issue F**: Update `reserveScheduledTopicAndCreateJob` to exclude topics whose normalized form matches any existing `articleTranslations.slug` (or `articles.slug` normalized), and exclude topics linked to jobs failed within 90 days.
2. **Fix Issue E**: Either add `triggerKind = 'scheduled'` filter to `listRetryableFailedJobs`, or document that manual jobs may also be retried by the daily cron.
3. **Fix Issue C**: Expand `revalidateAfterPublish` to include `/${loc}` (homepage), category routes if known, and `/sitemap.xml`.
4. **Add tests**: Unit/integration tests for scheduled controller flow, topic queue dedupe, retry endpoint, and auto-publish orchestration path.
5. **Consider**: Add `triggerKind` filter to retry endpoint if manual retry behavior is unwanted.

## Conclusion

Core architecture is solid. Tests, lint, typecheck, and build all pass. Two functional gaps need attention before this meets all acceptance criteria: (1) duplicate slug/topic dedupe in scheduled topic selection, and (2) incomplete ISR revalidation path coverage.
